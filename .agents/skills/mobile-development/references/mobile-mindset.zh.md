# 移动开发思维模式与思考方式

成功进行移动开发所必需的思维模式和决策框架。

## 移动开发十诫

### 1. 性能是基石，而非特性
- **现实**：如果加载时间超过 3 秒，70% 的用户会放弃使用应用。
- **思维模式**：从第一天就开始优化，而不是“以后再说”。
- **行动**：在编写代码之前设定性能预算。

### 2. 每一字节、每一毫秒都至关重要
- **现实**：移动设备是受限环境（电池、内存、网络）。
- **思维模式**：桌面的假设在这里不适用。
- **行动**：在真实设备上进行性能分析，而不是模拟器。

### 3. 默认离线优先 (Offline-First)
- **现实**：网络是不可靠的（电梯、隧道、飞机、信号差）。
- **思维模式**：为离线状态设计，在线时进行同步。
- **行动**：本地持久化优先，云端同步次之。

### 4. 用户语境 > 开发者环境
- **现实**：用户可能在火车上、行走中、单手操作或在强光下使用。
- **思维模式**：在现实场景中进行测试。
- **行动**：必须进行真机测试。

### 5. 了解平台特性，但不被平台锁定
- **现实**：iOS 和 Android 用户期望不同的交互模式。
- **思维模式**：尊重惯例，但保持逻辑的可移植性。
- **行动**：特定于平台的 UI，共享的业务逻辑。

### 6. 迭代，不求完美 (2024-2025 生存策略)
- **现实**：移动领域变化迅速。
- **思维模式**：发布、测量、改进的循环。
- **行动**：MVP（最小可行性产品）→ 用户反馈 → 迭代。

### 7. 设计之初即考虑安全与无障碍
- **现实**：这不仅是事后补充，而是核心需求。
- **思维模式**：从一开始就建立信任和包容性。
- **行动**：在每个冲刺 (Sprint) 中进行安全审计和无障碍测试。

### 8. 在真机上测试
- **现实**：模拟器在性能、电池和网络方面会“撒谎”。
- **思维模式**：模拟器用于速度，真机用于获取真相。
- **行动**：在 CI/CD 中集成真机设备农场 (Device Farms)。

### 9. 架构随复杂度扩展
- **现实**：过度设计会扼杀简单的应用。
- **思维模式**：从简单开始，需要时重构。
- **行动**：小型应用使用 MVVM，复杂度高时使用整洁架构 (Clean Architecture)。

### 10. 持续学习是生存之道
- **现实**：85% 的开发者使用 AI 工具 (2024)，框架不断演进。
- **思维模式**：拥抱变化，分配学习时间。
- **行动**：每周至少 1 小时用于学习新技术/模式。

## 移动端特有的限制与思考

### 小屏幕 (限制 → 设计参数)

**限制：**
- 5-7 英寸屏幕，拇指热区，胖手指问题（触控精度低）。

**思维转变：**
- 拥抱极简主义：“我们可以去掉什么？”
- 基于优先级的层级结构：最重要的操作置于显眼位置。
- 渐进式披露 (Progressive Disclosure)：将复杂性隐藏在层级之后。

**实践目标：**
- 最小触摸目标 44x44px (iOS)
- 最小触摸目标 48x48px (Android)
- 主要操作位于拇指可达范围内（屏幕底部 1/3）
- 底部导航栏最多 3-4 个项目

**决策示例：**
```
❌ 坏：在移动端展示 8 列数据表
✅ 好：卡片视图展示 3 个关键指标，点击“查看更多”获取详情
```

### 资源受限 (每一 KB/ms 都重要)

**限制：**
- 电池消耗、内存压力、热节流 (Thermal Throttling)。

**思维转变：**
- 在每一个决策中都保持资源意识。
- 优化前先测量（不要猜测）。
- 在低端设备上优雅降级。

**实践目标：**
- 典型屏幕内存占用 <100MB
- 每小时活跃使用电池消耗 <5%
- 初始下载 <50MB，总计 <200MB
- 60 FPS (每帧 16.67ms)

**决策示例：**
```
❌ 坏：加载列表中的全部 1000 个项目
✅ 好：虚拟化列表 (10 个可见项目 + 缓冲区)
```

### 间歇性连接 (离线优先)

**限制：**
- 网络不可靠：电梯、隧道、信号差、飞行模式。

**思维转变：**
- 本地优先的数据架构。
- 乐观 UI 更新 (Optimistic UI)。
- 同步冲突解决策略。

**实践方法：**
- **透写缓存 (Write-through cache)**：写入本地，后台同步。
- **混合同步**：推送 (实时) + 拉取 (定期)。
- **冲突解决**：基于时间戳的“最后写入胜出”或 CRDT。

**决策示例：**
```
❌ 坏：发布评论时显示加载转圈
✅ 好：立即显示评论（乐观更新），后台同步，处理冲突
```

## 平台特定思维

### iOS 心智模型

**哲学**：一致、精致、固执己见
- 用户期望 iOS 模式（标签栏、导航栏、滑动返回）。
- 设计审查会拒绝非标准 UI。
- “它就是好用 (It just works)”的期望 = 对崩溃零容忍。

**碎片化**：低
- 90%以上用户使用 iOS 16+ (2024)。
- 仅需测试约 50 种设备型号。
- 硬件规格可预测。

**设计思维**：
- 严格遵循人机界面指南 (HIG)。
- 原生导航模式不可协商。
- 重要操作要有触觉反馈。
- 尊重安全区域（刘海屏、灵动岛）。

**何时选择原生 iOS：**
- App Store 是主要收入渠道。
- 需要最新的 Apple 功能 (WidgetKit, Live Activities)。
- 目标是富裕用户群（iOS 用户支出高出 2.5 倍）。

### Android 心智模型

**哲学**：灵活、可定制、民主
- 用户期望 Material Design 但容忍差异。
- 极度碎片化 = 防御性编程。
- “返回键” = 根本性的导航期望。

**碎片化**：高
- 24,000+ 种设备型号。
- Android 6-14 均在活跃使用（跨越 8 年的操作系统版本）。
- 硬件规格范围广（512MB 到 12GB RAM）。

**设计思维**：
- 以 Material Design 3 为基准。
- 在低端设备上测试（最低 1GB RAM）。
- 尊重系统导航（手势 vs 三按钮）。
- 正确处理返回键。

**何时选择原生 Android：**
- 专注于全球市场（72% 市场份额）。
- 新兴市场（Android 占主导地位）。
- 企业/B2B（需要定制化）。

## 性能思维 (每一毫秒都重要)

### 关键指标 (用户感知)

| 指标 | 阈值 | 用户感知 |
|--------|-----------|-----------------|
| **启动时间** | <2s | 可接受 |
| **启动时间** | 2-3s | 明显延迟 |
| **启动时间** | >3s | 70% 用户放弃 |
| **屏幕加载** | <1s | 即时 (缓存) |
| **屏幕加载** | 1-3s | 可接受 (网络) |
| **屏幕加载** | >3s | 令人沮丧 |
| **动画** | 60 FPS | 流畅 |
| **动画** | 30-60 FPS | 明显卡顿 (Jank) |
| **动画** | <30 FPS | 无法使用 |

### 性能预算示例

**移动应用性能预算：**
```
启动时间
├─ 冷启动: <2s (目标 1.5s)
├─ 暖启动: <1s
└─ 热启动: <0.5s

屏幕加载
├─ 缓存数据: <500ms
├─ 网络数据: <2s
└─ 重计算: <3s

内存
├─ 典型屏幕: <100MB
├─ 重度屏幕 (图像): <150MB
└─ 峰值使用: <200MB

网络
├─ 初始包: <2MB
├─ 每屏幕: <500KB
└─ 图像: <200KB 每张

电池
├─ 活跃使用: <5% 每小时
├─ 后台: <1% 每小时
└─ 待机: <0.1% 每小时
```

### 优化决策树

**觉得慢吗？**
1. **先测量** (Xcode Instruments, Android Profiler)
2. **找瓶颈** (CPU, 内存, 网络, 磁盘 I/O)
3. **修复最大影响点** (二八法则)
4. **再次测量** (验证改进)

**常见罪魁祸首：**
- 主线程上的同步操作
- 未优化的图像（太大、格式错误）
- N+1 查询问题（循环中获取数据）
- 内存泄漏（循环引用、监听器）
- 无记忆化 (Memoization) 的重新渲染

## 移动开发工作流

### 迭代开发周期 (敏捷)

**冲刺结构 (2 周):**
```
第 1 周: 构建 + 测试
├─ 第 1-2 天: 设计 + 计划
├─ 第 3-4 天: 实现核心功能
└─ 第 5 天: 代码审查 + 测试

第 2 周: 润色 + 发布
├─ 第 6-7 天: 修复 Bug + 润色
├─ 第 8 天: QA 测试
├─ 第 9 天: 预发布部署 (Staging)
└─ 第 10 天: 生产发布 (分阶段)
```

**日常工作流：**
1. 拉取最新代码
2. 本地运行测试
3. 开发功能/修复
4. 编写/更新测试
5. 设备上本地测试
6. 代码审查
7. CI/CD 验证
8. 合并到开发分支

**CI/CD 影响：**
- 开发时间减少 20%
- 生产环境 Bug 减少 50%
- 部署速度快 3 倍

### 常见陷阱与避免

#### 1. 仅在模拟器上测试
**问题**：模拟器无法显示真实性能（电池、内存、网络）。
**解决方案**：每次发布前必须进行真机测试。
**影响**：40% 的 Bug 仅在真机上出现。

#### 2. 忽视平台惯例
**问题**：自定义导航会让用户感到困惑。
**解决方案**：遵循 iOS HIG 和 Material Design。
**影响**：非标准 UI 导致参与度降低 30%。

#### 3. 无离线处理
**问题**：网络故障 = 空白屏幕、错误。
**解决方案**：离线优先架构，缓存数据。
**影响**：50% 的用户每天都会遇到网络问题。

#### 4. 糟糕的内存管理
**问题**：内存泄漏 → 崩溃、性能差。
**解决方案**：理解 ARC/GC，定期进行性能分析。
**影响**：内存问题 = 第一大崩溃原因 (35%)。

#### 5. 硬编码凭证
**问题**：安全漏洞，API 密钥泄露。
**解决方案**：环境变量，安全存储。
**影响**：23% 的应用泄露敏感数据 (OWASP)。

#### 6. 无无障碍设计
**问题**：排除了 15% 以上的用户（残疾、情境性障碍）。
**解决方案**：VoiceOver/TalkBack 测试，语义标签。
**影响**：无障碍市场 = 全球 13 亿人。

#### 7. 过早优化
**问题**：浪费时间优化非瓶颈问题。
**解决方案**：先测量，优化最大影响点。
**影响**：80% 的性能问题 = 20% 的代码。

#### 8. 过度设计
**问题**：为简单的应用设计复杂的架构。
**解决方案**：从简单开始，需要时扩展。
**影响**：开发时间延长 3 倍，用户无收益。

#### 9. 跳过真机测试
**问题**：错过电池消耗、过热问题。
**解决方案**：CI/CD 中的设备农场，手动测试。
**影响**：25% 的性能问题是设备特定的。

#### 10. 不尊重电池
**问题**：后台处理耗尽电池。
**解决方案**：批量操作，尊重打盹模式 (Doze Mode)。
**影响**：电池消耗 = 第一大卸载原因。

## 渐进增强与优雅降级

### 渐进增强 (Build Up)

**策略**：从基准开始，为能力更强的设备进行增强。

**示例：图像加载**
```
基准 (所有设备):
├─ 立即显示占位符
├─ 加载低分辨率图像 (10KB)
└─ 平滑淡入显示

增强 (现代设备):
├─ 检查网络 (快 = 高分辨率)
├─ 检查内存 (充足 = 缓存)
└─ 渐进式 JPEG 渲染
```

**益处：**
- 适用于所有设备
- 在现代设备上获得最佳体验
- 不遗漏任何用户

### 优雅降级 (Strip Down)

**策略**：为最佳效果构建，针对限制进行降级。

**示例：动画**
```
最佳 (旗舰设备):
├─ 复杂的粒子效果
├─ 120 FPS 动画
└─ 视差滚动

降级 (预算设备):
├─ 简单的淡入淡出过渡
├─ 60 FPS 目标
└─ 禁用视差 (GPU 负载)
```

**检测：**
```javascript
// React Native
const isLowEndDevice =
  DeviceInfo.getTotalMemory() < 2000000000; // <2GB

if (isLowEndDevice) {
  // 禁用重型动画
  // 减少并发操作
  // 降低图像质量
}
```

**益处：**
- 针对所有硬件层级进行优化
- 防止低端设备崩溃
- 跨谱系的更好用户体验

## 原生 vs 跨平台决策框架

### 决策树

**Q1: 你需要 100% 的原生性能吗？**
- **是** → 原生 (Swift/Kotlin)
- **否** → 继续

**Q2: 团队对 JavaScript 熟悉吗？**
- **是** → React Native
- **否** → 继续

**Q3: 也需要桌面版或网页版吗？**
- **是** → Flutter
- **否** → 继续

**Q4: 需要复杂的动画或自定义 UI 吗？**
- **是** → Flutter
- **否** → React Native (更容易实现标准 UI)

**Q5: 有现有的代码库可以共享吗？**
- **React Web 应用** → React Native
- **无现有代码** → Flutter (更干净的起步)

### 混合方法 (两全其美)

**策略**：大多数功能跨平台，关键路径原生。

**架构示例：**
```
React Native / Flutter (90%)
├─ UI 和业务逻辑
├─ 标准功能
└─ API 集成

原生模块 (10%)
├─ 性能关键 (视频处理)
├─ 平台特定 (HealthKit, Android Auto)
└─ 第三方 SDK (支付, 分析)
```

**何时使用：**
- 最佳：利用跨平台速度 + 原生能力
- 复杂性：维护原生模块知识
- 团队：需要同时具备跨平台和原生开发者

## 架构决策

### 基于复杂度的架构选择

**简单应用 (1-5 个屏幕，基本 CRUD)**
- **架构**：MVVM (无整洁架构)
- **状态**：本地状态 (useState, setState)
- **理由**：过度设计增加复杂性且无益处

**中型应用 (5-20 个屏幕，中等逻辑)**
- **架构**：MVVM 配合清晰的分离
- **状态**：全局状态管理 (Zustand, Riverpod)
- **理由**：扩展性，无过度设计

**复杂应用 (20+ 屏幕，企业级逻辑)**
- **架构**：整洁架构 (领域层、数据层、表现层)
- **状态**：高级状态管理 + 依赖注入
- **理由**：可维护性和可测试性至关重要

### 架构演进

**从简单开始：**
```
v1.0: MVVM, 本地状态, 单模块
└─ 重点: 快速发布, 验证想法

v2.0: 需要时添加全局状态
└─ 触发点: 属性透传 (Props drilling) 变得痛苦

v3.0: 扩展时添加整洁架构
└─ 触发点: 团队增长, 功能倍增

v4.0: 如果合理，提取微服务
└─ 触发点: 独立部署需求
```

**关键原则：** 当痛苦 > 重构成本时才重构，而非之前。
